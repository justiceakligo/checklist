# Reqara Frontend Handoff: V2 Package Destinations and Connectors

API base:

```text
https://api.reqara.com
```

All routes below require dashboard auth and organization context unless stated otherwise.

## Product Boundary

Destinations are package handoff connectors. They should help an organization route an accepted package into an existing system after Reqara has completed the lifecycle:

```text
Collect -> Review -> Accept -> Package -> Route -> Prove delivery -> Archive
```

Reqara should not be presented as replacing SharePoint, OneDrive, Google Drive, SFTP, payroll, HRIS, EMR, CRM, or document-management systems. Reqara prepares a reviewed package, delivers it, records proof, and keeps an audit trail.

## Destination Types

Use these backend enum values:

```ts
type DestinationType =
  | "ManualDownload"
  | "Email"
  | "Webhook"
  | "Sftp"
  | "SharePoint"
  | "OneDrive"
  | "GoogleDrive"
  | "ApiPull"
  | "CsvExport"
  | "JsonExport";
```

Recommended UI labels:

```text
ManualDownload -> Manual handoff
Email -> Approved inbox
Webhook -> Webhook
Sftp -> SFTP
SharePoint -> SharePoint
OneDrive -> OneDrive
GoogleDrive -> Google Drive
ApiPull -> API pull
CsvExport -> CSV export
JsonExport -> JSON export
```

## What Is Real Now

`Email` sends a package-ready notification to configured inboxes. It does not attach submitted files.

`Webhook` sends signed package metadata to an HTTPS endpoint. It does not push file bytes.

`Sftp`, `SharePoint`, `OneDrive`, and `GoogleDrive` upload the true package ZIP bundle generated by the backend. The ZIP contains the report, metadata, answers JSON, audit summary, file manifest, and embedded clean submitted files when bundle limits allow.

`ManualDownload`, `ApiPull`, `CsvExport`, and `JsonExport` remain configurable/tracked handoff destinations. Use package export endpoints for actual files/data.

## UI Flow

Destination setup should be a wizard:

```text
1. Choose destination
2. Configure visible settings
3. Enter private credentials
4. Validate
5. Send test
6. Enable routing
```

Connected destination rows should show:

```text
Name
Type
Status
Default
Active
Last validated
Last validation status
Used by routing rules
Last delivery status
```

Use `POST /api/v2/destinations/{id}/validate` before enabling a destination. Use `POST /api/v2/destinations/{id}/test` before enabling automatic routing.

## Security UX Rules

Never show saved secrets back to the user.

For webhook destinations, show `generatedSigningSecret` once after creation if the backend generated it.

For SFTP, require the user to enter a pinned host key fingerprint. Do not allow "trust on first use" in production UI.

For Microsoft and Google destinations, prefer OAuth/admin consent flows later. The current backend supports server-side credentials and access tokens through `secretConfiguration`.

## Entitlements

Current feature gates:

```text
ManualDownload -> team workspace
Email -> automatic reminders
Webhook -> API and webhooks
Sftp -> custom workflows
SharePoint -> custom workflows
OneDrive -> custom workflows
GoogleDrive -> custom workflows
```

If backend returns `402 feature_not_available`, show an upgrade prompt.

## Create Destination

```http
POST /api/v2/destinations
Content-Type: application/json
```

Base request:

```json
{
  "name": "HR SharePoint folder",
  "type": "SharePoint",
  "configuration": {},
  "secretConfiguration": {},
  "status": "Active",
  "isDefault": false,
  "isActive": true
}
```

Secrets are encrypted by the backend and never returned.

## Approved Inbox

Use backend type `Email`.

Create:

```json
{
  "name": "HR approved inbox",
  "type": "Email",
  "configuration": {
    "recipients": ["hr@company.com"],
    "cc": ["ops@company.com"]
  },
  "status": "Active",
  "isDefault": false,
  "isActive": true
}
```

Validation rules:

```text
configuration.recipients must contain at least one email.
```

Test behavior:

```text
POST /api/v2/destinations/{id}/test
```

The backend sends a real package-ready test email through the configured email provider.

Create:

```json

  "name": "Payroll SFTP drop",
  "type": "Sftp",
  "configuration": {
    "remotePath": "/incoming/reqara",
    "fileNameTemplate": "{packageReference}-{recipientName}.zip"
  },
  "secretConfiguration": {
    "host": "sftp.company.com",
    "port": 22,
    "username": "reqara",
    "password": "secret-password",
    "hostKeySha256": "SHA256:abc123..."
  },
  "status": "Active",
  "isDefault": false,
  "isActive": true
}
```

Private key auth:

```json
{
  "host": "sftp.company.com",
  "port": 22,
  "username": "reqara",
  "privateKey": "-----BEGIN OPENSSH PRIVATE KEY-----\n...\n-----END OPENSSH PRIVATE KEY-----",
  "privateKeyPassphrase": "optional",
  "hostKeySha256": "SHA256:abc123..."
}
```

Validation rules:

```text
configuration.remotePath required
secretConfiguration.host required
secretConfiguration.username required
secretConfiguration.hostKeySha256 required
secretConfiguration.password or privateKey required
```

Test behavior:

```text
POST /api/v2/destinations/{id}/test
```

The backend creates a small test package bundle and uploads it to the configured remote path.

```json
  "isDefault": false,
  "isActive": true
```


## Validate Destination

```http
POST /api/v2/destinations/{destinationId}/validate
```

Success:

```json
{
  "id": "33d04e9f-45d5-48d4-ae20-8bb9e9afe2b1",
  "valid": true,
  "validatedAt": "2026-07-18T15:04:00Z"
}
```

Validation only checks local configuration and presence of stored secret config. It does not perform network delivery.

## Routing Rules

Attach a destination to a template:

```http
POST /api/v2/routing-rules
Content-Type: application/json
```

```json
{
  "templateId": "366bf827-3c2b-4384-b9f2-2ba179dd13f8",
  "destinationId": "33d04e9f-45d5-48d4-ae20-8bb9e9afe2b1",
  "trigger": "OnAcceptance",
  "sequence": 1,
  "isRequired": true,
  "isAutomatic": true,
  "configurationOverrides": {}
}
```

List routing rules:

```http
GET /api/v2/routing-rules?templateId=366bf827-3c2b-4384-b9f2-2ba179dd13f8
```

Automatic rules run after a submission is accepted and the package is created.

## Delivery Jobs

Get job:

```http
GET /api/v2/delivery-jobs/{jobId}
```

Retry:

```http
POST /api/v2/delivery-jobs/{jobId}/retry
```

Cancel:

```http
POST /api/v2/delivery-jobs/{jobId}/cancel
```

Statuses:

```text
Queued
Preparing
Sending
Succeeded
Failed
RetryScheduled
Cancelled
RequiresAttention
```

If a destination fails but has attempts remaining, backend schedules retry with `RetryScheduled` and `nextRetryAt`.

## Package Status Impact

Delivery jobs update package status:

```text
All delivery jobs succeeded -> Delivered
Some succeeded -> PartiallyDelivered
Any failed/requires attention and none succeeded -> DeliveryFailed
Pending jobs only -> RoutingPending
Manual handoff recorded -> HandoffConfirmed
```

## Frontend Notes

For `Sftp`, `SharePoint`, `OneDrive`, and `GoogleDrive`, call `validate`, then `test`, then allow the destination to be attached to automatic routing.

For setup screens, keep advanced credential entry behind an "Advanced connection details" section. Later OAuth UI can replace direct credential entry for Microsoft and Google while preserving the same destination records.

For package detail screens, show the delivery `externalReference` as proof only when it is not null. For SharePoint/OneDrive/Google Drive it may be a provider URL or provider object ID. For SFTP it is an `sftp://host:port/path/file.zip` reference.

For failed delivery, show `failureCode`, `failureMessage`, and a Retry action when status is not `Succeeded` or `Cancelled`.
